# Phase 99: Download UI and Colab Integration - Research

**Researched:** 2026-03-14
**Domain:** Astro component design, Google Colab GitHub URL integration, static asset linking, Git repository file management
**Confidence:** HIGH

## Summary

This phase adds two UI elements to each of the 10 case study pages -- a "Download Notebook" button linking to the build-time-generated ZIP file and an "Open in Colab" badge linking to Google Colab's GitHub URL scheme -- and commits all 10 `.ipynb` files to the repository so Colab can access them. The work is primarily frontend (Astro component creation) plus a build-time script to generate and commit the notebook files.

The case study pages are rendered by `src/pages/eda/case-studies/[...slug].astro`, which uses content from `src/data/eda/pages/case-studies/{slug}.mdx`. Each MDX file already imports and uses the `CaseStudyDataset` component (for CSV download and NIST source links). The new notebook UI elements should follow the same pattern -- either a new standalone `NotebookActions` component placed in the MDX files, or added directly to the page template `[...slug].astro`. Adding to the page template is strongly recommended because it avoids editing all 10 MDX files and keeps the notebook actions consistent across all case studies automatically.

The Google Colab GitHub URL format is `https://colab.research.google.com/github/{user}/{repo}/blob/{branch}/{path-to-notebook}.ipynb`. For this project, notebooks committed at `notebooks/eda/{slug}.ipynb` produce Colab URLs like `https://colab.research.google.com/github/PatrykQuantumNomad/PatrykQuantumNomad/blob/main/notebooks/eda/normal-random-numbers.ipynb`. The official Colab badge SVG is hosted at `https://colab.research.google.com/assets/colab-badge.svg`.

**Primary recommendation:** Create a `NotebookActions.astro` component that renders both the download button and Colab badge, driven by the case study slug. Add it to the `[...slug].astro` page template (not individual MDX files) so all case studies get it automatically. Use a build script or npm lifecycle hook to generate and commit `.ipynb` files to `notebooks/eda/`.

### Anti-Patterns to Avoid
- **Editing all 10 MDX files individually:** Use the page template instead for automatic coverage of all case studies
- **Hard-coding download URLs with full domain:** Use relative paths (`/downloads/notebooks/{slug}.zip`) so the links work in dev, preview, and production
- **Self-hosting the Colab badge SVG:** The official badge at `colab.research.google.com/assets/colab-badge.svg` is the standard; self-hosting adds maintenance burden
- **Generating notebooks at build time for Colab URLs:** Colab needs the files in the git repo, not in the built output. The `.ipynb` files must be committed to git.
- **Placing committed notebooks in `src/`:** Committed notebooks belong in a top-level `notebooks/` directory, not mixed with source code

## Common Pitfalls

### Pitfall 1: Colab URL Pointing to Non-Existent File
**What goes wrong:** Colab badge links work in theory but show "file not found" when clicked.
**Why it happens:** The `.ipynb` files were not committed to the git repo, or were committed to the wrong path, or the branch name in the URL is wrong.
**How to avoid:** Verify the exact URL pattern: `https://colab.research.google.com/github/PatrykQuantumNomad/PatrykQuantumNomad/blob/main/notebooks/eda/{slug}.ipynb`. Commit files to `notebooks/eda/` and verify they exist on GitHub before the Colab links go live.
**Warning signs:** 404 when clicking "Open in Colab" badge.

### Pitfall 2: Download Button 404 in Development Mode
**What goes wrong:** Download button links to `/downloads/notebooks/{slug}.zip` but returns 404 during `astro dev`.
**Why it happens:** ZIP files are generated at build time by the `astro:build:done` hook. They don't exist during development.
**How to avoid:** Accept this limitation -- download buttons will only work after `astro build`. Optionally add a `title` attribute or tooltip explaining this. Do NOT generate ZIPs during dev (it would slow down the dev server).
**Warning signs:** Development-only 404 on download links.

### Pitfall 3: handbook/ Files Not Available on GitHub for Colab Fallback
**What goes wrong:** When a user opens a notebook in Colab, the data loading cell tries to read the local `.DAT` file (which doesn't exist in Colab), falls back to the `githubRawUrl`, but that URL returns 404 because `handbook/` is in `.gitignore`.
**Why it happens:** The `githubRawUrl` values in the registry point to `raw.githubusercontent.com/.../handbook/datasets/RANDN.DAT`, but `handbook/` is gitignored and these files are not in the repo.
**How to avoid:** This is a known issue that should be addressed: either (a) commit the specific `.DAT` files the notebooks need to a non-gitignored location (e.g., `notebooks/eda/data/`), or (b) update the `githubRawUrl` values to point to the NIST source directly, or (c) bundle the `.DAT` files alongside the committed `.ipynb` files. Note: the ZIP downloads already include the `.DAT` files (Phase 98), so users who download the ZIP are fine. Only the "Open in Colab" path is affected.
**Warning signs:** `FileNotFoundError` followed by `urllib.error.HTTPError: HTTP Error 404` in Colab.

### Pitfall 4: Prose CSS Overriding Button Styles
**What goes wrong:** The download button or Colab badge inherits prose styling from the parent `div` with the `prose-foundations` class.
**Why it happens:** The case study page wraps `<Content />` in a div with extensive prose-like Tailwind selectors. If the NotebookActions component is placed inside this div, link colors, text sizes, and decorations may be overridden.
**How to avoid:** Place the `NotebookActions` component OUTSIDE the prose wrapper div, between the `<header>` and the prose `<div>` in `[...slug].astro`. Alternatively, use `!important` overrides (the existing `!text-white` and `!no-underline` pattern already used in CaseStudyDataset).
**Warning signs:** Buttons appear with wrong colors, underlines, or font sizes.

### Pitfall 5: Missing alt Text on Colab Badge Image
**What goes wrong:** Accessibility audit flags the Colab badge image as missing alt text.
**Why it happens:** External badge SVGs loaded via `<img>` need explicit `alt` attributes.
**How to avoid:** Always include `alt="Open in Colab"` on the badge image element.
**Warning signs:** aXe or Lighthouse accessibility warnings.

### Pitfall 6: Only 7 of 10 Notebooks Available Initially
**What goes wrong:** The NotebookActions component renders download/Colab buttons for all 10 case studies, but only 7 have notebooks (standard slugs from Phase 97).
**Why it happens:** Phase 100 hasn't been completed yet, so beam-deflections, random-walk, and ceramic-strength don't have buildable notebooks (they need advanced templates).
**How to avoid:** The component must check slug availability against `STANDARD_SLUGS` (or a similar whitelist) and only render buttons for slugs that have committed notebooks. Phase 100 will extend this list. Use an explicit allow-list, not the full `CASE_STUDY_REGISTRY` which has all 10.
**Warning signs:** 404 on download ZIP or "file not found" in Colab for the 3 advanced case studies.

### Notebook Generation Script
```javascript
// scripts/generate-notebooks.mjs
// Generates .ipynb files for committing to git (for Colab URLs)
import { mkdirSync, writeFileSync } from 'node:fs';
import { join } from 'node:path';

// Dynamic imports for TypeScript modules
const { buildStandardNotebook, STANDARD_SLUGS } = await import(
  '../src/lib/eda/notebooks/templates/standard.ts'
);

const outDir = join(process.cwd(), 'notebooks', 'eda');
mkdirSync(outDir, { recursive: true });

for (const slug of STANDARD_SLUGS) {
  const notebook = buildStandardNotebook(slug);
  const json = JSON.stringify(notebook, null, 1) + '\n';
  writeFileSync(join(outDir, `${slug}.ipynb`), json, 'utf-8');
}

console.log(`Generated ${STANDARD_SLUGS.length} notebooks in notebooks/eda/`);
```

**Deprecated/outdated:**

## Open Questions

1. **How to handle Colab data file access (handbook/ is gitignored)?**
   - What we know: The notebooks' data loading code uses `try/except FileNotFoundError` with a `githubRawUrl` fallback. But `handbook/` is in `.gitignore`, so those URLs return 404 on GitHub.
   - What's unclear: Whether to (a) commit specific .DAT files alongside notebooks, (b) update githubRawUrl to use NIST source URLs, or (c) accept this limitation since ZIP downloads include the data.
   - Recommendation: Commit the .DAT files alongside the notebooks in `notebooks/eda/data/` and update the `githubRawUrl` to point to `notebooks/eda/data/{file}.DAT` instead. This makes Colab actually work. Alternatively, if this is too large a change for Phase 99, document it as a known limitation and address it in Phase 100 or a follow-up.

2. **Should the 3 advanced slugs show a "coming soon" indicator?**
   - What we know: beam-deflections, random-walk, and ceramic-strength pages exist but won't have notebooks until Phase 100.
   - What's unclear: Whether to show nothing, show disabled buttons, or show a "notebook coming soon" message.
   - Recommendation: Show nothing (component returns null for non-available slugs). This is simplest and avoids broken UX. Phase 100 will automatically enable them when it adds the builders and committed notebooks.

3. **TypeScript/ESM compatibility for the notebook generation script**
   - What we know: The notebook builders are TypeScript modules in `src/lib/eda/notebooks/`. A script to generate `.ipynb` files for git commit needs to import these.
   - What's unclear: Whether to use `tsx` to run the script, or use Vitest as a runner, or use the Astro dev server's TypeScript support.
   - Recommendation: Use `npx tsx scripts/generate-notebooks.ts` (tsx is a TypeScript execution engine) or write the generation as a Vitest test that outputs files. The tsx approach is simpler.
