**GA-B001-missing-timeout.ts:**
- id: 'GA-B001', title: 'Missing timeout-minutes', severity: 'warning', category: 'best-practice'
- explanation: Jobs without timeout-minutes default to 360 minutes (6 hours). A hung job consumes runner resources for hours. Always set explicit timeout (typically 5-30 minutes for CI builds).
- fix: beforeCode shows job without timeout, afterCode adds `timeout-minutes: 30`
- check(): Use `forEachJobNode()`. For each job, `resolveKey(jobNode, 'timeout-minutes')`. If null, emit violation at job key position using `getNodeLine(jobKeyNode, ctx.lineCounter)`. Message: `Job '{jobName}' has no timeout-minutes. Default is 360 minutes (6 hours).`

**GA-B002-missing-concurrency.ts:**
- id: 'GA-B002', title: 'Missing concurrency group', severity: 'info', category: 'best-practice'
- explanation: Without a `concurrency:` group, multiple workflow runs for the same branch or PR execute simultaneously, wasting resources. A concurrency group with `cancel-in-progress: true` automatically cancels outdated runs.
- fix: beforeCode shows workflow without concurrency, afterCode adds `concurrency: { group: '${{ github.workflow }}-${{ github.ref }}', cancel-in-progress: true }`
- check(): `resolveKey(ctx.doc.contents, 'concurrency')`. If null, emit violation at line 1, column 1. Message: `Workflow has no concurrency group. Parallel runs may waste resources.`

**GA-B003-unnamed-step.ts:**
- id: 'GA-B003', title: 'Unnamed step', severity: 'info', category: 'best-practice'
- explanation: Steps with `run:` commands but no `name:` field are hard to identify in the GitHub Actions UI. Named steps improve log readability and debugging. Steps using `uses:` are self-documenting and excluded from this check.
- fix: beforeCode shows `run: npm test` without name, afterCode adds `name: Run tests`
- check(): Use `forEachJobNode()` + `forEachStepNode()`. For each step, check if it has a `run` key (via resolveKey) AND does NOT have a `name` key. If so, emit violation. Skip steps that have a `uses:` key (they are self-descriptive). Use the `run` key node for line position resolution.

**GA-B004-duplicate-step-name.ts:**
- id: 'GA-B004', title: 'Duplicate step name', severity: 'warning', category: 'best-practice'
- explanation: Multiple steps with the same name in a job make logs confusing. Each step name should be unique within its job to aid debugging and identification.
- fix: beforeCode shows two steps named "Run tests", afterCode makes them unique ("Run unit tests", "Run integration tests")
- check(): Use `forEachJobNode()` + `forEachStepNode()`. Collect step names per job (from resolveKey(stepNode, 'name')). Track in a Map<string, number> (name -> first occurrence index). On duplicate, emit violation at the duplicate's name node position. Message: `Step name '{name}' is duplicated in job '{jobName}'.`

**Create remaining 4 rule files:**

**GA-B005-empty-env.ts:**
- id: 'GA-B005', title: 'Empty env block', severity: 'info', category: 'best-practice'
- explanation: Empty `env:` blocks add noise to workflow files without setting any variables. Remove them or add the intended environment variables.
- check(): Check three levels: (1) top-level `env` on `ctx.doc.contents`, (2) each job's `env` via `forEachJobNode()`, (3) each step's `env` via `forEachStepNode()`. An env block is "empty" if `resolveKey` returns a node that is an empty Map (isMap && items.length === 0) or a null/empty Scalar. Use a helper function `isEmptyEnv(node)` for DRY. Emit violation at the `env` key node position.

**GA-B006-missing-conditional.ts:**
- id: 'GA-B006', title: 'Missing conditional on PR workflow', severity: 'info', category: 'best-practice'
- explanation: In workflows triggered only by `pull_request` or `pull_request_target`, per-job conditionals can prevent unnecessary work (e.g., skip deployment on draft PRs). Only flagged when the workflow has 2+ jobs.
- check(): First, determine if workflow triggers are PR-only: `resolveKey(ctx.doc.contents, 'on')`. If the `on` value is a Map, check if all keys are in Set('pull_request', 'pull_request_target'). If a Scalar, check if value matches. If not PR-only, return empty. Then, count jobs via `forEachJobNode()`. If fewer than 2 jobs, return empty. Finally, iterate jobs and flag any without an `if` key (via resolveKey). Message: `Job '{jobName}' has no 'if:' conditional in a PR-only workflow.`

**GA-B007-outdated-action.ts:**
- id: 'GA-B007', title: 'Outdated action version', severity: 'info', category: 'best-practice'
- explanation: Using outdated major versions of well-known GitHub Actions may miss important security fixes, performance improvements, and Node.js runtime updates. GitHub deprecated Node 16 action runners in 2024 and Node 20 follows in late 2026.
- Define `KNOWN_CURRENT_VERSIONS: Record<string, number>` constant with a `// Last verified: 2026-03-04` comment:
  ```
  actions/checkout: 4, actions/setup-node: 4, actions/setup-python: 5,
  actions/setup-java: 4, actions/setup-go: 5, actions/upload-artifact: 4,
  actions/download-artifact: 4, actions/cache: 4, actions/github-script: 7,
  github/codeql-action: 3
  ```
- Helper: `parseUsesVersion(usesValue: string): { action: string; major: number } | null` -- splits on last `@`, extracts major version from `/^v(\d+)/`.
- check(): Use `forEachUsesNode()`. For each uses value, call parseUsesVersion. If the action is in KNOWN_CURRENT_VERSIONS and the parsed major is 2+ versions behind current, emit violation. Message: `Action '${usesValue}' is outdated. Current major version is v${current}.`

**GA-B008-missing-continue-on-error.ts:**
- id: 'GA-B008', title: 'Missing continue-on-error on network step', severity: 'info', category: 'best-practice'
- explanation: Steps that fetch external data (curl, wget) can fail due to transient network issues. Adding `continue-on-error: true` prevents the entire workflow from failing due to a non-critical external request.
- Conservative keyword list: `['curl ', 'curl\n', 'wget ', 'wget\n', 'gh api']` (with trailing space/newline to avoid matching substrings like "curling"). Also match these at the start of the run value.
- check(): Use `forEachRunNode()`. For each run value, check if it contains any keyword from the list. If yes, check if the step has `continue-on-error` set (need to access the step node -- modify approach to use `forEachJobNode()` + `forEachStepNode()` instead, then for each step check for `run` key, test keywords, and check for `continue-on-error` key). Emit violation at the run node position. Message: `Step uses '${keyword}' without continue-on-error. Network failures may break the workflow.`
