Create `src/lib/tools/gha-validator/rules/style/` directory with 4 rule files:

**GA-F001-jobs-not-alphabetical.ts:**
- id: 'GA-F001', title: 'Jobs not alphabetically ordered', severity: 'info', category: 'style'
- explanation: Alphabetical job ordering makes large workflow files easier to navigate and produces more predictable diffs in version control. While GitHub Actions does not require any particular ordering, consistency improves readability.
- fix: beforeCode `jobs:\n  deploy:\n    ...\n  build:\n    ...`, afterCode `jobs:\n  build:\n    ...\n  deploy:\n    ...`
- check(): Get `jobs` node via `resolveKey(ctx.doc.contents, 'jobs')`. If not isMap, return empty. Collect job key names and their AST key nodes. If fewer than 2 jobs, return empty (single job is trivially sorted). Create sorted copy with `localeCompare(a, b, undefined, { sensitivity: 'base' })`. Find first index where order differs. Report ONE violation at that job key position. Message: `Jobs are not in alphabetical order. Found '{actual}' where '{expected}' was expected.` Break after first mismatch -- one violation is sufficient to signal the issue.

**GA-F002-inconsistent-quoting.ts:**
- id: 'GA-F002', title: 'Inconsistent uses: quoting', severity: 'info', category: 'style'
- explanation: Mixing quoting styles (plain, single-quoted, double-quoted) for `uses:` values reduces consistency. Pick one style and use it throughout the workflow.
- fix: beforeCode shows mixed `uses: actions/checkout@v4` and `uses: 'actions/setup-node@v4'`, afterCode shows consistent unquoted style
- check(): Use `forEachUsesNode()` but access the raw AST node, not just the string value. For each uses value node, check `isScalar(node)` and read `(node as Scalar).type`:
  - `Scalar.PLAIN` -> 'plain'
  - `Scalar.QUOTE_SINGLE` -> 'single'
  - `Scalar.QUOTE_DOUBLE` -> 'double'
  - anything else -> skip

  Collect `{ style, node }` entries. If all same style or fewer than 2 entries, return empty. Determine the majority style (most common). Flag all nodes using a non-majority style. Emit ONE violation at the first non-majority node position. Message: `Inconsistent uses: quoting. This value uses {minority} but {majority} is used elsewhere.`

**GA-F003-long-step-name.ts:**
- id: 'GA-F003', title: 'Step name exceeds 80 characters', severity: 'info', category: 'style'
- explanation: Very long step names reduce readability in the GitHub Actions UI and log output. Keep step names concise (under 80 characters) to improve scannability.
- fix: beforeCode shows a step name over 80 chars, afterCode shows a shorter version
- check(): Use `forEachJobNode()` + `forEachStepNode()`. For each step, get `name` via `resolveKey(stepNode, 'name')`. If isScalar and String value length > 80, emit violation at the name node position. Message: `Step name is ${length} characters (max recommended: 80).`

**GA-F004-missing-workflow-name.ts:**
- id: 'GA-F004', title: 'Missing workflow name', severity: 'info', category: 'style'
- explanation: Workflows without a top-level `name:` field display as the filename in the GitHub Actions UI, which is less descriptive. Adding a name improves the workflow list readability.
- fix: beforeCode shows `on: push\njobs:`, afterCode shows `name: CI\non: push\njobs:`
- check(): `resolveKey(ctx.doc.contents, 'name')`. If null, emit violation at line 1, column 1. Message: `Workflow has no top-level 'name:' field.`

```typescript
/**
 * Master rule registry for the GitHub Actions Workflow Validator.
 *
 * Aggregates all custom lint rules from category subdirectories.
 * Phase 76: 10 security rules
 * Phase 77: 18 actionlint metadata + 8 best practice + 4 style rules
 */

import type { GhaLintRule } from '../types';
import { securityRules } from './security';
import { bestPracticeRules } from './best-practice';
import { styleRules } from './style';
import { actionlintMetaRules } from './semantic';

/** All custom lint rules with check() methods (run in Pass 1). */
export const allGhaRules: GhaLintRule[] = [
  ...securityRules,
  ...bestPracticeRules,
  ...styleRules,
];

/** All documented rules (custom + actionlint metadata) for static page generation. */
export const allDocumentedGhaRules: GhaLintRule[] = [
  ...allGhaRules,
  ...actionlintMetaRules,
];

/** Look up a rule by ID across all documented rules. Returns undefined if not found. */
export function getGhaRuleById(id: string): GhaLintRule | undefined {
  return allDocumentedGhaRules.find((r) => r.id === id);
}
```

Note: `allGhaRules` has 22 entries (10+8+4). `allDocumentedGhaRules` has 40 entries (22+18). `getGhaRuleById` searches the full 40-rule set so actionlint metadata rules are also discoverable.
