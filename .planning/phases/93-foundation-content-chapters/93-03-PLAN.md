MCP servers expose tools that Claude Code can call just like its built-in tools.

3. "Architecture Overview" -- Brief explanation of the client-server architecture: Claude Code is the MCP client, external tools run as MCP servers. Embed McpArchitectureDiagram after the explanation. Add a paragraph explaining the diagram shows the topology with local servers (stdio) and remote servers (HTTP) with transport badges.

4. "Transport Modes" -- Explain the 3 transport options:
   - **stdio** (default for local): Server runs as a child process. Claude Code communicates via stdin/stdout. Best for local tools. Most common.
   - **HTTP** (recommended for remote): Server runs remotely. Uses Streamable HTTP protocol. Supports authentication. The recommended transport for remote servers.
   - **SSE** (DEPRECATED): Server-Sent Events. Was the original remote transport. Explicitly deprecated in favor of HTTP. Still works but will be removed.
   Show the --transport flag syntax via CodeBlock.

5. "Adding MCP Servers" -- Show the claude mcp add command via CodeBlock (lang="bash"):
   ```
   claude mcp add my-server --transport stdio -- npx my-mcp-server
   ```
   Explain the command structure: name, transport flag, double-dash separator, then the command to run. Show variations:
   - With environment variables: --env KEY=value
   - With scope: --scope project (adds to .mcp.json instead of ~/.claude.json)
   - With HTTP: --transport http --url https://server.example.com
   - With headers for auth: --header "Authorization: Bearer TOKEN"

6. "Server Scopes" -- Explain the 3 scopes:
   - Local (default): Stored in ~/.claude.json. Only available to you on this machine.
   - Project: Stored in .mcp.json in project root. Shared with team via git.
   - User: Stored in user settings. Available across all projects.
   Explain when to use each: local for personal tools, project for team-shared tools (databases, CI), user for tools you use everywhere.

7. ".mcp.json Format" -- Show the .mcp.json file structure via CodeBlock (lang="json", title=".mcp.json"):
   ```json
   {
     "mcpServers": {
       "database": {
         "command": "npx",
         "args": ["@mcp/postgres", "postgresql://localhost/mydb"],
         "env": { "DB_PASSWORD": "${DB_PASSWORD}" }
       }
     }
   }
   ```
   Explain environment variable expansion with ${VAR} syntax. Note this file should be committed to git for team sharing (but use env var expansion for secrets).

8. "OAuth Authentication" -- Explain the OAuth flow for MCP servers that require authentication:
   - Run /mcp command to see server status
   - For servers requiring auth, Claude Code opens a browser for the OAuth flow
   - Tokens are stored and refreshed automatically
   Show the /mcp command via CodeBlock (lang="text").

9. "Tool Search" -- Explain the automatic tool search feature:
   - When context usage exceeds ~10% and MCP tools are not actively needed, Claude Code unloads them to save context
   - Tool search automatically reloads relevant tools when needed
   - Can be controlled via ENABLE_TOOL_SEARCH env var
   Explain this is transparent to the user -- tools appear when Claude needs them.

10. "MCP Resources" -- Briefly explain that MCP servers can expose resources (not just tools):
    - Resources are data sources like files, database tables, API endpoints
    - Access via @ mentions in prompts
    - Example: @database/users to reference a table

11. "Managed MCP" -- Explain enterprise controls:
    - managed-mcp.json for organization-wide server configuration
    - allowedMcpServers / deniedMcpServers for controlling which servers users can connect
    - Managed servers cannot be removed by users
    Show example managed-mcp.json via CodeBlock.

12. "Troubleshooting" -- Common issues and solutions:
    - /mcp status -- check which servers are connected, errored, or disconnected. Show via CodeBlock.
    - Server not starting: check command path, verify package is installed, check logs
    - Token limits: MAX_MCP_OUTPUT_TOKENS controls how much output an MCP tool can return
    - Timeout: servers must respond within the configured timeout
    Show troubleshooting commands via CodeBlock.

13. "Best Practices" -- 5-7 bullets: start with stdio for local development, use .mcp.json for team-shared servers, use env var expansion for secrets in .mcp.json, prefer HTTP over SSE for remote servers, keep MCP server count manageable (each consumes context), use /mcp status to verify connections before relying on tools.

14. "Further Reading" -- Links to official docs (mcp page) and cross-link to Chapter 7 (custom skills) as the next chapter in the learning path.

- stdio is the default for local servers
- Tool search activates at ~10% context threshold
- .mcp.json uses ${VAR} syntax for env var expansion
- Do NOT describe MCP server development (creating servers) -- only connecting/configuring
