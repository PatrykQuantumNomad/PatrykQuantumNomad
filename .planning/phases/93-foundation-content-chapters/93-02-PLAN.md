   - User (~/.claude/settings.json) -- personal preferences across all projects
   - Project (.claude/settings.json) -- shared with team via git

4. "Environment Variables" -- List the key environment variables:
   - ANTHROPIC_MODEL -- override default model
   - CLAUDE_CODE_EFFORT -- set effort level
   - ANTHROPIC_API_KEY -- API key (for direct API usage)
   - CLAUDE_CODE_OUTPUT_DIR -- custom output directory
   - DISABLE_AUTOUPDATER -- disable auto-updates
   Show how to set them in shell profile via CodeBlock (lang="bash", title="~/.zshrc").

5. "Why Sandboxing Matters" -- Explain the security rationale: Claude Code runs bash commands and modifies files. Two scenarios: protecting against accidental damage (rm -rf outside project) and protecting against prompt injection in untrusted codebases.

6. "Sandbox Modes" -- Explain the two modes:

   - macOS: Seatbelt (sandbox-exec) restricts file system access at kernel level
   - Linux: bubblewrap (bwrap) provides namespace-based isolation
   Explain sandbox.filesystem.allowWrite for specifying writable paths. Show example with prefix notation (// for absolute, ~ for home, ./ for relative). Via CodeBlock.

8. "Network Isolation" -- Explain proxy-based network restrictions:
   - All outbound traffic routed through a local proxy
   - allowedDomains setting controls which domains can be reached
   - Default: allows npm registry, PyPI, and other package managers
   Show allowedDomains example via CodeBlock.

9. "Status Line Customization" -- Explain the custom status line feature:
   - statusLine setting points to a shell script
   - The script receives session JSON on stdin and outputs status text
   - Displayed at the bottom of the terminal during Claude Code sessions
   Show example shell script via CodeBlock (lang="bash", title="~/.claude/status.sh") that extracts model and cost from session JSON.

10. "Best Practices" -- 5-7 bullets: use project settings for team conventions (committed to git), use local settings for personal overrides (gitignored), enable sandbox for untrusted codebases, start with regular sandbox before auto-allow, keep allowedDomains minimal, use status line to track costs.
