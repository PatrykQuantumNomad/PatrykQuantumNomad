  <behavior>
    GRAPH-01 (Three node types):
    - extractGhaGraphData({ on: 'push', jobs: { build: { steps: [{ name: 'Checkout' }] } } }, [])
      -> nodes include { id: 'trigger-push', type: 'trigger', label: 'push' }
      -> nodes include { id: 'job-build', type: 'job', label: 'build' }
      -> nodes include { id: 'step-build-0', type: 'step', label: 'Checkout', parentJobId: 'job-build', stepIndex: 0 }

    - on: as string ('push'), array (['push', 'pull_request']), and object ({ push: { branches: ['main'] } }) all produce trigger nodes

    GRAPH-02 (Job dependency edges + cycle detection):
    - jobs: { test: {}, deploy: { needs: 'test' } } -> edge { source: 'job-test', target: 'job-deploy', edgeType: 'needs' }
    - jobs: { a: { needs: 'b' }, b: { needs: 'a' } } -> hasCycle: true, cycle edges flagged with isCycle: true
    - needs: as string ('test') and array (['test', 'lint']) both handled

    GRAPH-03 (Trigger-to-job edges):
    - Triggers connect to entry-point jobs (jobs with no needs: dependency)
    - jobs: { test: {}, deploy: { needs: 'test' } } with on: 'push' -> trigger edge from trigger-push to job-test only (not to job-deploy)

    GRAPH-04 (Steps as sequential nodes):
    - Steps have parentJobId linking to their parent job
    - Steps have stepIndex for sequential ordering (0, 1, 2...)
    - Step label: step.name ?? step.uses ?? step.run?.slice(0, 30) ?? 'Step {i+1}'

    GRAPH-05 (Violation status):
    - Job with error-severity violation -> violationStatus: 'error'
    - Job with warning but no error -> violationStatus: 'warning'
    - Job with no violations -> violationStatus: 'clean'
    - Violation matching uses job ID substring matching (violation ruleId or job key in violation context)

    Edge cases:
    - Empty jobs: block -> returns only trigger nodes
    - No on: block -> returns only job/step nodes, no trigger edges
    - Job with empty steps: [] -> job node with no child step nodes
    - Missing jobs entirely -> returns { nodes: [], edges: [], hasCycle: false }
  </behavior>
  <implementation>
    1. Export interfaces: GhaGraphNode (id, type, label, parentJobId?, violationStatus, stepIndex?), GhaGraphEdge (id, source, target, edgeType, label?, isCycle?), GhaGraphData (nodes, edges, hasCycle)

    2. extractGhaGraphData(json, violations) function:
       a. Parse on: block (string/array/object) -> trigger nodes
       b. Parse jobs: block -> job nodes with violation status
       c. Parse steps: arrays -> step nodes with parentJobId and stepIndex
       d. Build needs: edges between jobs (handle string and array forms)
       e. Connect triggers to entry-point jobs (jobs with no needs:)
       f. Run cycle detection (port Kahn's algorithm from compose graph-builder.ts)
       g. Mark cycle edges with isCycle: true
       h. Return GhaGraphData

    3. computeJobViolationStatus() helper: aggregates violations at job level. For simplicity, match violations to jobs by checking if any violation message or context references the job key. If no line-range data available, aggregate all violations across the workflow and assign 'clean' to jobs with no matching violations.

    4. detectJobCycles() helper: Kahn's algorithm ported from compose-validator/graph-builder.ts detectCycles(). Operates on Map<string, string[]> (jobId -> needs[]).
  </implementation>
</feature>
