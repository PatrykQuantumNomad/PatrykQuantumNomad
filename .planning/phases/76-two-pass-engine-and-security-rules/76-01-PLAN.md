<objective>
Create the two-pass validation engine with unified violation types, deduplication logic, and actionlint error mapping.

Purpose: This is the orchestration layer that ties together Phase 75's schema validator and WASM Worker into a single pipeline. Pass 1 runs synchronously (schema + custom rules), Pass 2 merges async WASM results with deduplication. Without this, the two validation engines operate independently with no unified output.

Output: engine.ts with runPass1/mergePass2 functions, extended types.ts with GhaLintRule/GhaUnifiedViolation/GhaRuleContext, engine tests, and ghaViolations store atom.
</objective>

Create `src/lib/tools/gha-validator/engine.ts` following the compose-validator engine.ts pattern but with two-pass architecture:

1. **Imports:** parseGhaWorkflow from parser, validateGhaSchema + categorizeSchemaErrors from schema-validator, types (GhaUnifiedViolation, GhaRuleContext, GhaRuleViolation, ActionlintError, GhaSeverity, GhaCategory).

2. **ACTIONLINT_KIND_MAP:** Record mapping all 16 actionlint error kinds to `{ ruleId: string, severity: GhaSeverity, category: GhaCategory }`. Use the exact mapping from the research:
   - syntax-check -> GA-L001 (error, semantic)
   - expression -> GA-L002 (error, semantic)
   - job-needs -> GA-L003 (error, semantic)
   - matrix -> GA-L004 (error, semantic)
   - events -> GA-L005 (error, semantic)
   - glob -> GA-L006 (warning, semantic)
   - action -> GA-L007 (error, semantic)
   - runner-label -> GA-L008 (error, semantic)
   - shell-name -> GA-L009 (error, semantic)
   - id -> GA-L010 (error, semantic)
   - credentials -> GA-L011 (error, security)
   - env-var -> GA-L012 (warning, semantic)
   - permissions -> GA-L013 (error, security)
   - workflow-call -> GA-L014 (error, semantic)
   - shellcheck -> GA-L017 (warning, actionlint)
   - pyflakes -> GA-L018 (warning, actionlint)
   Fallback for unknown kinds: GA-L000 (warning, actionlint)

3. **GhaEngineResult interface:** `{ violations: GhaUnifiedViolation[], rulesRun: number, rulesPassed: number, pass: 1 | 2 }`

4. **toUnified(v: GhaRuleViolation, severity: GhaSeverity, category: GhaCategory): GhaUnifiedViolation** -- Maps base violation to unified format by spreading fields and adding severity + category.

5. **mapActionlintError(err: ActionlintError): GhaUnifiedViolation** -- Looks up kind in ACTIONLINT_KIND_MAP, falls back to GA-L000.

6. **runPass1(rawText: string, customRules?: GhaLintRule[]): GhaEngineResult** -- Synchronous. Steps:
   a. Parse YAML via parseGhaWorkflow(rawText)
   b. Convert parse errors to unified format (severity: 'error', category: 'schema')
   c. If json exists: run schema validation via validateGhaSchema + categorizeSchemaErrors, convert to unified
   d. If json exists and customRules provided: create GhaRuleContext, iterate rules, call rule.check(ctx), convert results to unified using rule.severity and rule.category, count rulesPassed
   e. Sort violations by line then column
   f. Return GhaEngineResult with pass: 1

   The `customRules` parameter defaults to empty array (no import from rules/ directory -- that circular dependency is avoided by passing rules in). This keeps engine.ts independent of which rules exist.

7. **mergePass2(pass1Violations: GhaUnifiedViolation[], actionlintErrors: ActionlintError[]): GhaEngineResult** -- Steps:
   a. Build `occupied` Set from pass1Violations: `${v.line}:${v.column}`
   b. Map each actionlintError via mapActionlintError, skip if key in occupied, else add to pass2 array and occupied set
   c. Merge: [...pass1Violations, ...pass2Violations]
   d. Sort by line then column
   e. Return with pass: 2

3. **runPass1 with valid YAML:** Returns empty violations for a valid workflow

9. **mergePass2 no overlap:** Both violations at different positions are kept
10. **mergePass2 sorting:** Output sorted by line then column
11. **All unified violations have required fields:** ruleId, message, line, column, severity, category
