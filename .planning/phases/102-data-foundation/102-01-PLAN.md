  <name>Task 1: Create Zod schemas and TypeScript types</name>
  <files>src/lib/ai-landscape/schema.ts</files>
  <action>
Create `src/lib/ai-landscape/schema.ts` with Zod schemas following the established project pattern (see `src/lib/beauty-index/schema.ts`).

Import: `import { z } from 'astro/zod';` (project convention -- NOT bare `zod`).

Define these schemas:

1. `exampleSchema` -- for grouped product/framework examples:
   - `name: z.string()` -- display name (e.g., "GPT-4o")
   - `description: z.string()` -- brief description of what this product/framework is

2. `aiNodeSchema` -- for concept nodes:
   - `id: z.string()` -- URL-safe slug, also used as collection ID (e.g., "large-language-models"). REQUIRED by file() loader.
   - `name: z.string()` -- display name (e.g., "Large Language Models (LLM)")
   - `slug: z.string()` -- URL slug for /ai-landscape/[slug]/ routes (same value as id)
   - `cluster: z.string()` -- cluster ID (e.g., "genai", "dl", "ml")
   - `parentId: z.string().nullable()` -- parent concept node ID, null for top-level cluster root nodes
   - `simpleDescription: z.string().min(50)` -- layperson-level explanation, ~100-150 words
   - `technicalDescription: z.string().min(50)` -- Wikipedia-level explanation, ~100-150 words
   - `whyItMatters: z.string()` -- 1-2 sentence real-world connection
   - `examples: z.array(exampleSchema).default([])` -- grouped product/framework examples
   - `dotNodeId: z.string()` -- original DOT node identifier for traceability

3. `edgeSchema` -- for relationships:
   - `source: z.string()` -- source node slug
   - `target: z.string()` -- target node slug
   - `label: z.string()` -- human-readable relationship label from DOT (e.g., "subset of")
   - `type: z.enum([...])` -- categorized edge type. Include these values: 'hierarchy' (for "subset of" backbone), 'includes' (for "includes fields", "types", "architectures", "paradigms", "technique", "variants"), 'enables' (for "enables", "powers"), 'example' (for "e.g."), 'relates' (for "relates to", "informs", "combined with DL", "trains via", "extends to", "uses"), 'progression' (for "Potential Progression"), 'characterizes' (for "characterized by"), 'aspires' (for "aspires to", "aims for", "steps toward"), 'applies' (for "paradigm applied", "built with"), 'standardizes' (for "standardizes", "enables interop", "used by")

4. `clusterSchema` -- for cluster definitions:
   - `id: z.string()` -- cluster identifier (e.g., "ai", "ml", "nn")
   - `name: z.string()` -- display name (e.g., "Artificial Intelligence (AI)")
   - `color: z.string()` -- light mode hex color from DOT bgcolor
   - `darkColor: z.string()` -- dark mode equivalent hex color
   - `parentClusterId: z.string().nullable()` -- parent cluster for nesting (null for top-level)

Export all schemas and their inferred types:
```typescript
export type AiNode = z.infer<typeof aiNodeSchema>;
export type Edge = z.infer<typeof edgeSchema>;
export type Cluster = z.infer<typeof clusterSchema>;
export type Example = z.infer<typeof exampleSchema>;
```

Also add a helper function for downstream phases:
```typescript
export function getNodeRelationships(nodeSlug: string, edges: Edge[]): {
  parents: Edge[];
  children: Edge[];
  related: Edge[];
} {
  return {
    parents: edges.filter(e => e.target === nodeSlug),
    children: edges.filter(e => e.source === nodeSlug),
    related: edges.filter(e =>
      (e.source === nodeSlug || e.target === nodeSlug) &&
      !['hierarchy', 'includes'].includes(e.type)
    ),
  };
}
```

**Note on two-file approach:** CONTEXT.md says "single JSON file with all nodes, edges, and cluster definitions." The two-file split (nodes.json + graph.json) is a technical accommodation of Astro's `file()` loader, which requires a flat homogeneous JSON array and cannot handle mixed entity types. This preserves the intent of a canonical data model while working within the framework's constraints. See RESEARCH.md line 210 for detailed justification.

The file structure:
```json
{
  "clusters": [...],
  "edges": [...]
}
```
