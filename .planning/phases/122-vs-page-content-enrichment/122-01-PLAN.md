  <name>Task 1: Implement vs-content.ts lib with three-seam paragraph assembly</name>
  <files>src/lib/beauty-index/vs-content.ts</files>
  <action>
Create `src/lib/beauty-index/vs-content.ts` as a pure TypeScript module. Export the types `VsDimensionEntry`, `VsCodeFeatureEntry`, `VsFaqEntry`, `VsCrossLink`, `VsContent` (shapes per RESEARCH Example 1) and the function `buildVsContent(langA: Language, langB: Language, allLangs: Language[]): VsContent`.

Imports required:
- `DIMENSIONS, type Dimension` from `./dimensions`
- `JUSTIFICATIONS` from `../../data/beauty-index/justifications`
- `CODE_FEATURES` from `../../data/beauty-index/code-features`
- `type Language` from `./schema`

Implement each sub-concern as an internal helper (all module-private except buildVsContent):

**1. Dimension ordering (CONTEXT locked decision):** Build an array of `VsDimensionEntry` for all 6 DIMENSIONS, then sort by `Math.abs(b.delta) - Math.abs(a.delta)` descending. Tie-break on `dim.key` lexicographic so ordering is stable.

**2. Per-dimension prose assembly (Pattern 1 from RESEARCH):** For each dimension, build a 4-6 sentence paragraph from three seams:
  - Per-language data: `JUSTIFICATIONS[langA.id][dim.key]` and `JUSTIFICATIONS[langB.id][dim.key]` (HTML-safe strings — preserve `<em>`/`<code>` tags, do NOT strip).
  - Per-pair connective clause: picked deterministically from a pool keyed on `(dim.key, deltaBucket)` where `deltaBucket` is one of `'tie' | 'narrow' | 'standard'` based on `|delta|` thresholds (0 / 1 / >=2). Each `(dim.key, bucket)` must have >=3 connective options; select by `hash(langA.id + langB.id + dim.key) % pool.length`. Store the pool as a module-level const `CONNECTIVE_POOLS: Record<DimKey, Record<DeltaBucket, string[]>>`. Clauses contain `{winner}` / `{loser}` / `{both}` placeholders that the renderer fills in.
  - Per-pair verdict clause: a one-liner stating who wins. For `delta === 0`: "Both score {scoreA} — this is one dimension where {langA.name} and {langB.name} genuinely agree." For `Math.abs(delta) === 1`: "{winner} edges {loser} by a single point on {dim.name}; the practical difference is slim but real." For `Math.abs(delta) >= 2`: "{winner} wins {dim.name} by {|delta|} points — {strengthWord}." where `strengthWord` varies from a pool of 4-6 options keyed on the bucket and dimension.
  - Closer clause: one sentence tying the dimension back to the pair's character. Drawn from a small pool keyed on `(dim.key, winner-paradigm-cluster)` with >=2 options per key.

Final paragraph structure: `<p>{verdict} {winnerJustification} {connective} {loserJustification} {closer}</p>`. The order of the two justifications is determined by who wins on THIS dimension (winner first), so the same pair reads differently across its 6 dimension paragraphs as the winner flips.

**3. Verdict hero paragraph (Pattern 2):** 2-3 sentences.
  - Sentence 1: overall-score framing. Compute `totalA = sum(scores for 6 dims)`, `totalB`, `gap = totalA - totalB`, `dimsWonByA = count of dims where langA leads`. Template: "`{higher} scores {totalWinner}/60 against {lower}'s {totalLoser}/60, leading in {dimsWon} of 6 dimensions.`"
  - Sentence 2: axis-cluster framing. Group dimensions into 4 clusters — aesthetic (phi, lambda), mathematical (omega), human (psi, gamma), design (sigma). Per cluster, determine winner by cluster-sum. Template varies based on how many clusters each language owns. If langA owns >=3 clusters: "`{langA.name} dominates the {clusterNames} axes.`" If even split: "`{langA.name} owns {clusterListA} while {langB.name} leads in {clusterListB}.`"
  - Sentence 3 (conditional): pair's distinguishing tension. Only emit if the two languages split across (psi winner != gamma winner). Template pool of 3 options keyed on the cluster-split signature.

The verdict must be >=2 sentences and must mention both language names.

**4. Code feature selection (Pattern 3 — differ-most, Claude's Discretion decision):** For every feature in `CODE_FEATURES` where both `feature.snippets[langA.id]?.code` and `feature.snippets[langB.id]?.code` are defined non-empty strings:
  - Compute `differenceScore = lineCountDelta * 0.3 + bigramJaccardDistance * 0.5 + keywordDensityDelta * 0.2` where:
    - `lineCountDelta = Math.abs(snipA.split('\n').length - snipB.split('\n').length) / 10` (capped at 1)
    - `bigramJaccardDistance = 1 - jaccard(charBigrams(snipA), charBigrams(snipB))`
    - `keywordDensityDelta = abs((reservedTokens(snipA)/tokens(snipA)) - (reservedTokens(snipB)/tokens(snipB)))`
  - Sort descending, take top 3. For each, produce `VsCodeFeatureEntry` with `featureName`, `caption = feature.description || feature.name`, and snippet objects containing `{ lang: snippets[langId].lang, label: snippets[langId].label, code: snippets[langId].code }`.

Fallback: if `<3` both-defined features exist (won't happen per coverage probe, but guard), return all available. Target is 2-3 per CONTEXT.

**5. FAQ derivation (Pattern 4 — 3 questions, score-driven, Claude's Discretion decision):**
  - Q1: "Which is easier to learn, {langA.name} or {langB.name}?" Answer derived from psi scores (winner on psi is the easier answer), 40-80 words, plain text, reuses phrasing from psi justifications stripped of HTML.
  - Q2: "Is {langA.name} or {langB.name} better for {useCase}?" — `useCase` comes from a DIM_TO_USE_CASE lookup keyed on `topDeltaDim.key`: lambda->"readable code", phi->"visually clean syntax", omega->"algorithm-heavy code", psi->"developer happiness", gamma->"long-lived codebases", sigma->"principled design". Answer uses the top-delta dim's prose stripped of HTML.
  - Q3: "Should I pick {langA.name} or {langB.name} in 2026?" Answer uses total-score gap + tier context: "`{langA.name} lands in the {langA.tier} tier at {totalA}/60; {langB.name} in the {langB.tier} tier at {totalB}/60. {recommendation}`" where recommendation is pulled from a pool of 3-5 options keyed on the gap-bucket (small/medium/large).

Answer strings must be plain text. Strip HTML via `.replace(/<[^>]+>/g, '')` before embedding. Each answer 40-80 words. Include a helper `stripHtml(s: string): string`.

**6. Cross-link selection (Pattern 5 — exactly 4 links):**
  - Reverse pair: `{ label: '{langB.name} vs {langA.name}', href: '/beauty-index/vs/{langB.id}-vs-{langA.id}/', placement: 'footer' }`.
  - Shared-language pair: from `allLangs`, find all pairs `(langA, X)` and `(X, langB)` where X != langA && X != langB. Rank by `Math.abs(totalScore(X) - totalScore(langA))` descending (prefer pairs with a score gap that DIFFERS from the current pair's gap, so the link provides editorial contrast). Pick the top candidate. Label = "{langA.name} vs {X.name}" or "{X.name} vs {langB.name}". Placement: 'inline'.
  - Paradigm-adjacent pair: group allLangs by `.paradigm` bucket. From langA's paradigm bucket (excluding langA, langB), pick the language X with highest total score; pair X with the next-highest language in a DIFFERENT bucket. Label = "{X.name} vs {Y.name}". Placement: 'footer'.
  - Single-language back-link: `{ label: langA.name, href: '/beauty-index/{langA.id}/', placement: 'inline' }`.

All anchor labels are comparison names or single-language names ONLY — no descriptive prose (CONTEXT locked decision). Determinism: selection is a pure function of langA.id, langB.id, and allLangs order.

**7. Word count sanity field:** Compute `wordCount` by joining all prose + verdict + faq answers + captions, stripping HTML, splitting on whitespace. This is a sanity-check field, NOT authoritative — the VS-07 check operates on built HTML.

All connective pools, verdict pools, closer pools, strengthWord pools, DIM_TO_USE_CASE map, Q3 recommendation pools, and cluster definitions MUST be module-level `const` (parsed once, not rebuilt per call). Pool sizes must be >=3 per key (>=2 for closer pools) to keep connective-collision low. NO hand-rolled randomness — all pool selection is deterministic via a small `hash(string): number` helper (e.g., 32-bit FNV-1a).

DO NOT use AI generation, do NOT make runtime network calls, do NOT modify JUSTIFICATIONS or CODE_FEATURES. Purely read-only consumption of existing data.
