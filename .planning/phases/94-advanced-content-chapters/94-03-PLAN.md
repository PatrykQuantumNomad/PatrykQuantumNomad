**Sections (use h2 headings):**

1. "What You Will Learn" -- 2-3 sentence overview: security model, managed settings deployment, governance controls (permissions, hooks, MCP, plugins), sandbox enforcement, authentication, vulnerability scanning as capability pattern.

2. "Security Model Overview" -- Claude Code's security is built on a permission-based architecture. Every potentially dangerous action requires explicit approval. The model is designed with defense-in-depth: built-in protections, configurable permissions, and enterprise governance layers. Cross-reference Chapter 3 (Models, Costs & Permissions) for the permission system basics. This chapter focuses on the enterprise administration layer.

3. "Managed Settings" -- Explain that managed settings are the enterprise deployment mechanism. They override all user and project settings.

   **5 delivery mechanisms:**
   - **Server-managed**: Fetched from Anthropic's servers for authenticated enterprise users
   - **macOS MDM**: Distributed via Apple MDM profiles
   - **Windows registry**: Set via Group Policy or direct registry keys (HKLM)
   - **File-based**: JSON file at a well-known path on the filesystem
   - **Windows HKCU**: Per-user registry settings (lower priority than HKLM)

   Show a CodeBlock example (lang="json", title="managed-settings.json") with a representative enterprise configuration including permission rules, hook controls, and sandbox settings.

4. "Settings Precedence" -- Explain the hierarchy:
   managed (highest) > CLI flags > local (.claude/settings.local.json) > project (.claude/settings.json) > user (~/.claude/settings.json)
   Managed settings always win. This is intentional -- it prevents developers from weakening enterprise security controls. The /status command shows which settings are active and their source.

5. "Permission Governance" -- allowManagedPermissionRulesOnly: when true, only managed permission rules apply. User/project rules are ignored. This ensures consistent security posture across the organization. Show a CodeBlock example of managed permission rules that allow specific tools and deny others.

6. "Hook Governance" -- allowManagedHooksOnly: when true, blocks user, project, local, and plugin hooks. Only managed hooks run. This prevents developers from installing hooks that bypass security controls.
   - allowedHttpHookUrls: whitelist of HTTP endpoints hooks can call
   - httpHookAllowedEnvVars: which environment variables hooks can access
   Show a CodeBlock example (lang="json") of hook governance configuration.

7. "MCP Server Governance" -- Controls which MCP servers can be connected:
   - allowManagedMcpServersOnly: when true, only org-approved MCP servers connect
   - allowedMcpServers: explicit allowlist of server names/patterns
   - deniedMcpServers: blocklist of prohibited servers
   - managed-mcp.json: file-based MCP server configuration
   Show a CodeBlock example of MCP governance settings.

8. "Plugin Governance" -- Controls plugin installation sources:
   - strictKnownMarketplaces: array of approved plugin sources (7 source types: github, npm, etc.)
   - blockedMarketplaces: array of blocked plugin sources
   - pluginTrustMessage: custom message shown when a user tries to install a non-approved plugin
   Show a CodeBlock example with strictKnownMarketplaces allowing only a corporate GitHub org.

9. "Sandbox Enforcement" -- Two isolation dimensions:
   - **Filesystem isolation**: allowWrite (paths the agent can write to), denyWrite (blocked write paths), denyRead (blocked read paths). Uses glob patterns.
   - **Network isolation**: allowedDomains (whitelist of domains for HTTP requests), allowManagedDomainsOnly (only managed domain list applies).
   - allowUnsandboxedCommands: when false, prevents running commands outside the sandbox
   Show a CodeBlock example (lang="json") of a full sandbox configuration restricting both filesystem and network access.

10. "Authentication Controls" -- Two settings for enterprise auth:
    - forceLoginMethod: restrict to "claudeai" (Claude.ai consumer) or "console" (Anthropic Console/API)
    - forceLoginOrgUUID: require authentication to a specific organization
    Brief section, show CodeBlock with both settings.

11. "Vulnerability Scanning as a Capability" -- IMPORTANT: "Vulnerability scanning" is NOT a standalone named feature in Claude Code. It is a capability pattern enabled by combining subagents (Chapter 9) and skills (Chapter 7). Show how to create a security-reviewer subagent (CodeBlock with AGENT.md example from research) that runs git diff, analyzes changes for security issues, and reports findings. Also mention that managed hooks can enforce security checks automatically via PreToolUse hooks on Write/Edit tools.

12. "Prompt Injection Protections" -- Explain the built-in protections:
    - Command blocklist for known dangerous patterns
    - Context-aware analysis that detects suspicious instructions in fetched content
    - Isolated fetch context (content fetched from URLs is marked as untrusted)
    Brief section covering defense-in-depth approach.

13. "Monitoring and Compliance" -- Briefly cover:
    - OpenTelemetry integration for telemetry export
    - otelHeadersHelper for custom header injection
    - companyAnnouncements for broadcasting messages to all Claude Code users in the org

14. "Verifying Configuration with /status" -- The /status command shows active settings per layer, configuration sources, and which managed settings are in effect. Recommend running /status after deploying managed settings to verify they took effect.

15. "Best Practices" -- 5-7 bullets: start with permissive settings and tighten gradually, use allowManagedPermissionRulesOnly for critical security controls, test managed settings on a pilot group first, use sandbox network isolation to prevent data exfiltration, keep allowedHttpHookUrls narrow, document your managed settings for team visibility, use /status to audit configuration.

16. "Further Reading" -- Links to official docs (code.claude.com/docs/en/security, code.claude.com/docs/en/settings, code.claude.com/docs/en/plugins) and back-link to Chapter 3 for permission basics.

**Critical accuracy rules:**
- "Vulnerability scanning" is a CAPABILITY PATTERN, not a named feature. Frame it as combining subagents + skills + hooks.
- There are exactly 5 managed settings delivery mechanisms
- Settings precedence is: managed > CLI > local > project > user
- allowManagedPermissionRulesOnly, allowManagedHooksOnly, allowManagedMcpServersOnly are the 3 governance lockdown flags
- Plugin governance uses strictKnownMarketplaces (7 source types)
- Do NOT invent security features or settings not in official docs
- Cross-reference Chapter 3 (permission basics) and Chapter 9 (subagents for vulnerability scanning pattern)
