Each skill is a SKILL.md file containing YAML frontmatter + markdown body. Skills can be invoked via slash commands, auto-invoked by Claude, or both. Note that .claude/commands/ files still work but skills (SKILL.md) are the recommended approach.

3. "SKILL.md Anatomy" -- Show a complete SKILL.md example via CodeBlock (lang="yaml", title="SKILL.md") demonstrating all key frontmatter fields and markdown body. 4. "Frontmatter Reference" -- Document all 10 optional frontmatter fields in a descriptive format (not a table that is hard to read in MDX):
   - **name**: Human-readable name (defaults to filename)
   - **description**: What the skill does (shown in /help and slash command menu)
   - **argument-hint**: Placeholder text for the slash command argument (e.g. "[issue-number]")
   - **disable-model-invocation**: When true, only the user can invoke via slash command (Claude cannot auto-invoke)
   - **user-invocable**: When false, only Claude can invoke (not available as slash command). Default true.
   - **allowed-tools**: Restrict which tools the skill can use (e.g. "Read, Grep, Glob")
   - **model**: Override the model used (e.g. "sonnet", "haiku")
   - **context**: Set to "fork" to run in a subagent with isolated context
   - **agent**: Which subagent to use when context is "fork" (e.g. "Explore")
   - **hooks**: Skill-scoped hook configuration (same format as settings.json hooks)
   Show a CodeBlock with a skill that uses several of these fields together.

5. "Where Skills Live" -- 4 storage locations with their scope and discovery:
   - Enterprise managed: Deployed via managed settings (admins control)
   - Personal: ~/.claude/skills/ (available in all your projects)
   - Project: .claude/skills/ (shared via git with team)
   - Plugin: Installed from plugin marketplaces
   Explain automatic discovery: nested .claude/skills/ directories in subdirectories are found automatically (monorepo support). Show a directory tree example via CodeBlock (lang="text").

6. "Invocation Modes" -- Three modes with clear explanations:
   - **Slash command** (default): User types /skill-name in chat. Set disable-model-invocation: true for user-only.
   - **Auto-invocation** (default): Claude decides to use the skill based on context. Set user-invocable: false for Claude-only.
   - **Both** (default behavior): Both user and Claude can invoke. This is the default when neither flag is set.
   Show CodeBlock examples for each mode's frontmatter configuration.

7. "String Substitutions" -- Explain the 5 string substitutions available in skill markdown body:
   - $ARGUMENTS: Full argument string passed by the user
   - $ARGUMENTS[N] or $N: Nth argument (0-indexed)
   - ${CLAUDE_SESSION_ID}: Current session identifier
   - ${CLAUDE_SKILL_DIR}: Absolute path to the skill's directory (for relative file references)
   Show a CodeBlock example of a skill using $ARGUMENTS and ${CLAUDE_SKILL_DIR}.

8. "Supporting Files and Dynamic Injection" -- Skills can reference supporting files in the same directory. Show a directory structure example with a skill and its templates/examples. Explain the !`command` syntax for dynamic context injection -- shell commands whose output is injected into the skill prompt at invocation time. Show a CodeBlock example (lang="markdown", title=".claude/skills/deploy/SKILL.md") using !`git branch --show-current` to inject the current branch.

9. "Running Skills in Subagents" -- When context: fork is set, the skill runs in an isolated subagent. This prevents the skill from consuming the parent session's context window. Combine with agent field to use a specific subagent (e.g. Explore for read-only analysis). Show a CodeBlock example of a fork+agent skill.

10. "Bundled Skills" -- List the built-in skills: /simplify (reduce code complexity), /batch (run across multiple files), /debug (systematic debugging), /loop (iterative refinement), /claude-api (Anthropic API integration). Brief one-line description for each.

11. "Restricting Skill Access" -- Permission rules can control which skills are allowed. Skill(name) permission type can be set to allow/deny. Show a CodeBlock example of denying a specific skill in settings.

12. "Best Practices" -- 5-7 bullets: keep skills focused on one task, write clear descriptions for discoverability, use allowed-tools to limit scope, test skills with simple inputs first, use context:fork for expensive operations, prefer project skills (.claude/skills/) for team sharing.

13. "Further Reading" -- Link to official docs (code.claude.com/docs/en/skills) and cross-link to Chapter 8 for hooks (which can be embedded in skills).
