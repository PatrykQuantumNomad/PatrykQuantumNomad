    - **plan**: Read-only mode (no file writes or command execution)

3. "What Are Agent Teams?" -- Agent teams enable multiple Claude Code instances to work together on a shared task list with direct communication. Unlike subagents (Chapter 9) which are single-session delegation, agent teams are multi-session coordination with independent Claude Code instances running in parallel.

4. "Enabling Agent Teams" -- Show how to enable via environment variable and via settings.json. CodeBlock (lang="bash") for export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1. CodeBlock (lang="json", title="settings.json") for the settings.json approach.

5. "Team Architecture" -- Embed AgentTeamsDiagram after a brief intro. Explain the 4 components:
   - **Lead**: The original Claude Code session that creates and manages the team. The lead is always fixed (cannot be changed mid-session).
   - **Teammates**: Independent Claude Code instances spawned by the lead. Each has its own context window, tools, and permissions.
   - **Shared task list**: A central list of tasks with states (pending, in-progress, completed) and dependency tracking. Tasks are claimed via file-lock to prevent duplicate work.
   - **Mailbox**: Communication channel for sending messages between teammates and the lead. Supports message-one and broadcast-all patterns.

6. "Starting a Team" -- Teams start from a natural language request. Either Claude proposes spawning teammates, or the user explicitly requests "create a team." The lead decomposes the work into tasks and spawns teammates. Show an example prompt via CodeBlock (lang="text").

7. "Display Modes" -- Two modes:
   - **In-process**: All teammates share the terminal. Use Shift+Down to cycle between teammate views. Default mode.
   - **Split-pane**: Each teammate gets its own terminal pane via tmux or iTerm2. Requires tmux or iTerm2 to be available.
   Show how to configure in settings.

8. "Task Management" -- Explain task states: pending (waiting), in-progress (claimed by a teammate), completed (done). Dependencies: tasks can depend on other tasks -- a dependent task stays pending until its prerequisites are completed. File-lock claiming prevents two teammates from working on the same task. Show a conceptual task list example via CodeBlock (lang="text").

9. "Teammate Communication" -- Three communication patterns:
   - **Message one**: Send a targeted message to a specific teammate
   - **Broadcast all**: Send a message to all teammates
   - **Automatic idle notifications**: TeammateIdle events trigger when a teammate finishes its work
   Communication is asynchronous -- teammates process messages when they reach a natural pause point.

10. "Agent Teams vs Subagents" -- Comparison to help readers choose the right tool. Key differences:
    - **Communication**: Subagents report to parent only. Teammates can message each other.
    - **Coordination**: Subagents have no shared state. Teams share a task list.
    - **Context**: Subagents share parent context. Teammates have independent context windows.
    - **Token cost**: Subagents use parent's allocation. Teams use significantly more tokens (scales with team size).
    - **Persistence**: Subagents end with parent. Teams end together when all tasks complete.
    Present as a markdown comparison table.

11. "Best Practices" -- 5-7 bullets: keep teams small (3-5 teammates), decompose work into clear independent tasks, use dependency tracking for sequential work, monitor token usage (teams are expensive), use in-process mode for quick coordination and split-pane for long-running work, end team sessions cleanly.

12. "Known Limitations" -- MUST include all verified limitations:
