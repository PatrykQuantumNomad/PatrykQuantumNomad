<OpeningStatement>Your codebase is full of code that no one understands, no one owns, and no one can safely change. AI did not create this problem. It poured gasoline on a fire that was already burning.</OpeningStatement>

<TldrSummary>

- **AI coding assistants have accelerated a pre-existing crisis.** Codebases are filling with code that no one understands, no one owns, and no one can safely change -- and the tools we celebrate for their productivity are making it worse.
- **The numbers are stark.** A 4x increase in code clones, refactoring collapsing from 25% to less than 10% of all changes, and AI-assisted developers scoring 17% lower on comprehension.
- **This essay introduces "The Dark Code Spectrum,"** a 5-dimension framework for measuring how much of your codebase has gone dark: Clone Density, Ownership Vacuum, Comprehension Decay, Refactoring Deficit, and Vulnerability Surface.
- **The only defense is deliberate illumination.** Measure it, own it, refactor it -- or accept the consequences when the code you cannot read becomes the code that fails in production.

</TldrSummary>

Every engineering team has a number they track. Velocity. Sprint points. Deployment frequency. But almost no one tracks the one metric that predicts whether the codebase will still be maintainable in two years: how much of it has gone dark[^1].
