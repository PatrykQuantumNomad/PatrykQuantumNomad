## What You Will Learn

This chapter covers the SKILL.md system for creating reusable, shareable instruction sets that extend Claude Code's capabilities. You will learn the anatomy of a SKILL.md file, all ten frontmatter configuration fields, the three invocation modes (slash command, auto-invocation, and both), the four storage locations where skills can live, string substitutions for dynamic content, the dynamic injection syntax for shell command preprocessing, and the bundled skills that ship with Claude Code.

By the end of this chapter, you will be able to create custom skills tailored to your workflow and share them with your team.

## What Are Skills?

Skills are reusable instruction sets that extend what Claude Code can do. Each skill is a SKILL.md file that contains YAML frontmatter for configuration and a markdown body with instructions that Claude reads and follows when the skill is invoked. Skills follow the Agent Skills open standard.

You can invoke a skill in three ways: as a slash command you type in chat, automatically when Claude determines it is relevant to your request, or both. Skills can restrict which tools Claude uses, override the model, run in isolated subagents, and even embed their own hook configurations.

Skills replace the older `.claude/commands/` directory. Command files still work for backward compatibility, but skills are the recommended approach -- they support richer configuration through frontmatter, automatic discovery in nested directories, and supporting files alongside the skill definition.

## SKILL.MD Anatomy

Every skill is a single SKILL.md file with two parts: YAML frontmatter between `---` delimiters and a markdown body below.

<CodeBlock
  code={`---
name: fix-issue
description: Fetch a GitHub issue and implement the fix
argument-hint: "[issue-number]"
allowed-tools: Read, Grep, Glob, Bash, Write, Edit
model: sonnet
---

You are a senior developer fixing a GitHub issue.

1. Fetch the issue details using: gh issue view $ARGUMENTS
2. Read the relevant source files identified in the issue
3. Implement the fix with minimal changes
4. Write or update tests to cover the fix
5. Run the test suite to verify nothing is broken
6. Summarize what you changed and why`}
  lang="yaml"
  title="SKILL.md"
  caption="A complete skill with frontmatter configuration and markdown instructions."
/>

The **frontmatter** (everything between the `---` markers) configures how the skill behaves: its name, description, which tools it can use, and how it can be invoked. All frontmatter fields are optional -- a SKILL.md with only a markdown body is valid.

The **markdown body** (everything after the closing `---`) is the instruction set that Claude reads and follows when the skill runs. Write it as clear, step-by-step instructions. Claude treats the body as a system-level prompt scoped to the skill's execution.

## Frontmatter Reference

SKILL.md supports ten optional frontmatter fields. None are required -- defaults apply when fields are omitted.

**name** -- Human-readable name for the skill. If omitted, the filename or directory name is used. This name appears in `/help` output and the slash command menu.

<CodeBlock
  code={`---
name: deploy-staging
---`}
  lang="yaml"
  title="Setting a skill name"
/>

**description** -- A brief explanation of what the skill does. Shown in `/help` and the slash command completion menu. Write a clear description so users (and Claude) understand when to use the skill.

**argument-hint** -- Placeholder text that appears after the slash command name in the completion menu. Tells the user what arguments the skill expects. For example, `[issue-number]` or `[file-path] [output-format]`.

**disable-model-invocation** -- When set to `true`, only the user can invoke this skill via its slash command. Claude cannot auto-invoke it. Use this for skills that should only run when explicitly requested, such as deployment scripts or destructive operations. Default is `false`.

**user-invocable** -- When set to `false`, the skill is not available as a slash command. Only Claude can invoke it based on context. Use this for internal utility skills that Claude should use automatically but users should not call directly. Default is `true`.

**allowed-tools** -- A comma-separated list of tools the skill is permitted to use. This restricts the skill's capabilities to only the tools you specify. For example, `Read, Grep, Glob` creates a read-only skill that cannot write files or run commands.

**model** -- Override the model used when this skill runs. Accepts model shortnames like `sonnet`, `haiku`, or `opus`. Useful for running cheaper models on simple tasks or ensuring the most capable model for complex operations.

**context** -- Set to `fork` to run the skill in an isolated subagent. The subagent gets its own context window, preventing the skill from consuming the parent session's context. See the Running Skills in Subagents section below.

**agent** -- Which subagent to use when `context` is set to `fork`. For example, `Explore` runs the skill with the built-in read-only Explore subagent. Only meaningful when `context: fork` is also set.

**hooks** -- Skill-scoped hook configuration using the same format as `settings.json` hooks. Hooks defined here only apply when this skill is running. See [Hooks & Lifecycle Automation](/guides/claude-code/hooks/) for the full hook configuration format.

Here is a skill that combines several frontmatter fields.

<CodeBlock
  code={`---
name: security-review
description: Audit source code for common security vulnerabilities
argument-hint: "[directory-or-file]"
disable-model-invocation: true
allowed-tools: Read, Grep, Glob
model: opus
context: fork
agent: Explore
---

You are a senior security engineer performing a code audit.

Analyze the code at $ARGUMENTS for:
- SQL injection vulnerabilities
- Cross-site scripting (XSS) vectors
- Authentication and authorization flaws
- Hardcoded secrets or credentials
- Insecure deserialization
- Path traversal vulnerabilities

For each finding, report:
1. The file and line number
2. The vulnerability type and severity
3. A recommended fix with code example`}
  lang="yaml"
  title=".claude/skills/security-review/SKILL.md"
  caption="A security review skill: user-only invocation, read-only tools, runs in an isolated Explore subagent."
/>

## Where Skills Live

Skills can be stored in four locations, each with a different scope and audience.

**Enterprise managed** skills are deployed through managed settings by organization administrators. Individual developers cannot modify or remove them. Use this for organization-wide standards like coding conventions, compliance checks, or approved deployment procedures.

**Personal** skills live in `~/.claude/skills/` and are available in every project you work on. Use this location for skills that match your personal workflow -- your preferred code review checklist, your debugging methodology, or your documentation template.

**Project** skills live in `.claude/skills/` at the root of your repository. They are committed to Git and shared with your entire team. This is the recommended location for team-specific skills -- everyone who clones the repository gets the same skill set.

**Plugin** skills are installed from plugin marketplaces. They extend Claude Code through community-contributed or organization-published skill packages.

<CodeBlock
  code={`my-project/
  .claude/
    skills/
      deploy/
        SKILL.md           # Team deployment skill
        templates/
          k8s-manifest.yaml
      code-review/
        SKILL.md           # Team code review skill
  packages/
    auth/
      .claude/
        skills/
          auth-debug/
            SKILL.md       # Package-specific skill (auto-discovered)
  src/
    ...`}
  lang="text"
  title="Project skill directory structure"
  caption="Skills in nested .claude/skills/ directories are discovered automatically (monorepo support)."
/>

Claude Code discovers skills automatically by scanning for `SKILL.md` files in `.claude/skills/` directories at any depth in your project. In a monorepo with multiple packages, each package can define its own skills and they will all be available when Claude Code runs from the project root.

## Invocation Modes

Skills support three invocation modes that control who can trigger them.

**Slash command invocation** is the default for user-facing skills. The user types `/skill-name` followed by optional arguments in the chat prompt. To make a skill user-only (preventing Claude from auto-invoking it), set `disable-model-invocation: true` in the frontmatter.

<CodeBlock
  code={`---
name: deploy
description: Deploy the current branch to staging
disable-model-invocation: true
---

Deploy steps here...`}
  lang="yaml"
  title="User-only slash command"
  caption="Setting disable-model-invocation: true prevents Claude from auto-invoking this skill."
/>

**Auto-invocation** lets Claude decide to use the skill based on context. When Claude determines that a skill is relevant to the current task, it invokes it automatically without the user typing a slash command. To make a skill Claude-only (hiding it from the slash command menu), set `user-invocable: false`.

<CodeBlock
  code={`---
name: test-helper
description: Automatically run relevant tests after code changes
user-invocable: false
---

Test running steps here...`}
  lang="yaml"
  title="Claude-only auto-invocation"
  caption="Setting user-invocable: false removes this skill from the slash command menu."
/>

**Both** (the default) allows both the user and Claude to invoke the skill. When neither `disable-model-invocation` nor `user-invocable: false` is set, the skill appears in the slash command menu and Claude can also invoke it automatically when relevant.

<CodeBlock
  code={`---
name: refactor
description: Refactor code for clarity and maintainability
---

Refactoring steps here...`}
  lang="yaml"
  title="Default: both user and Claude can invoke"
  caption="No invocation flags means both slash command and auto-invocation are enabled."
/>

## String Substitutions

The markdown body of a skill supports five string substitutions that inject dynamic values at invocation time.

**$ARGUMENTS** is replaced with the full argument string the user passes after the slash command. If the user types `/fix-issue 42 --priority high`, then `$ARGUMENTS` becomes `42 --priority high`.

**$ARGUMENTS[N]** or **$N** is replaced with the Nth argument (0-indexed). For `/fix-issue 42 --priority high`, `$0` is `42`, `$1` is `--priority`, and `$2` is `high`.

**$&#123;CLAUDE_SESSION_ID&#125;** is replaced with the current session identifier. Useful for creating session-specific log files or temporary directories.

**$&#123;CLAUDE_SKILL_DIR&#125;** is replaced with the absolute path to the directory containing the SKILL.md file. Use this for referencing supporting files that live alongside the skill (templates, configuration files, examples).

<CodeBlock
  code={`---
name: scaffold
description: Create a new component from the team template
argument-hint: "[component-name]"
---

Create a new React component named $ARGUMENTS using the template
at \${CLAUDE_SKILL_DIR}/templates/component.tsx as a starting point.

The component should be placed in src/components/$0/.
Copy the template, rename it, and update all placeholder names.
Also create a test file using \${CLAUDE_SKILL_DIR}/templates/component.test.tsx.`}
  lang="markdown"
  title=".claude/skills/scaffold/SKILL.md"
  caption="$ARGUMENTS passes the component name; CLAUDE_SKILL_DIR resolves to the skill's directory for template access."
/>

## Supporting Files and Dynamic Injection

Skills can include supporting files in the same directory as the SKILL.md. These files -- templates, configuration snippets, example outputs, reference documentation -- provide additional context that Claude can read when executing the skill.

<CodeBlock
  code={`.claude/skills/deploy/
  SKILL.md                    # The skill definition
  templates/
    k8s-deployment.yaml       # Kubernetes manifest template
    k8s-service.yaml          # Service template
  examples/
    staging-deploy.md         # Example of a successful deploy
  scripts/
    validate-manifest.sh      # Validation script`}
  lang="text"
  title="Skill with supporting files"
  caption="Supporting files live alongside SKILL.md and are referenced via CLAUDE_SKILL_DIR."
/>

For dynamic context that changes between invocations, SKILL.md supports the **!&#96;command&#96;** syntax. When Claude loads the skill, it runs the shell command and injects the output into the skill prompt. This lets your skill adapt to the current state of the repository, environment, or system.

<CodeBlock
  code={`---
name: deploy
description: Deploy current branch to the appropriate environment
argument-hint: "[environment]"
---

Deploy the current project to the $ARGUMENTS environment.

Current branch: !\`git branch --show-current\`
Last commit: !\`git log --oneline -1\`
Uncommitted changes: !\`git status --short\`

Read the deployment configuration at \${CLAUDE_SKILL_DIR}/templates/k8s-deployment.yaml
and customize it for the $0 environment.

If there are uncommitted changes, warn the user before proceeding.`}
  lang="markdown"
  title=".claude/skills/deploy/SKILL.md"
  caption="The !&#96;command&#96; syntax injects live shell output into the skill prompt at invocation time."
/>

When this skill runs, the three `!`command`` blocks are replaced with the actual output of `git branch --show-current`, `git log --oneline -1`, and `git status --short`. Claude sees the current branch name, latest commit, and working tree status as part of its instructions.

## Running Skills in Subagents

When a skill sets `context: fork` in its frontmatter, it runs in an isolated subagent with its own context window. This is valuable for two reasons: the skill does not consume context from your main session, and it can run without interfering with your ongoing conversation.

Combine `context: fork` with the `agent` field to specify which subagent runs the skill. The built-in `Explore` subagent uses the Haiku model and has read-only tools, making it ideal for analysis tasks that should not modify files.

<CodeBlock
  code={`---
name: analyze-deps
description: Analyze project dependencies for vulnerabilities and updates
context: fork
agent: Explore
---

Analyze this project's dependency files (package.json, requirements.txt,
go.mod, or similar) and report:

1. Total number of direct and transitive dependencies
2. Any dependencies with known security advisories
3. Dependencies that are more than 2 major versions behind
4. Unused dependencies (declared but not imported)

Read dependency lock files for accurate version information.
Do not modify any files.`}
  lang="yaml"
  title="Skill running in a forked Explore subagent"
  caption="context: fork isolates the skill; agent: Explore restricts it to read-only analysis."
/>

The subagent runs to completion and returns its results to the parent session. Your main conversation continues uninterrupted, and the skill's work does not reduce your available context window. This pattern is especially useful for expensive analysis tasks that read many files.

## Bundled Skills

Claude Code ships with several built-in skills that are available in every session without any configuration.

**/simplify** reduces code complexity. Point it at a function, class, or file and it will refactor for clarity -- extracting helper functions, simplifying conditionals, reducing nesting depth, and improving naming.

**/batch** runs an operation across multiple files. Describe a change once and it applies the transformation consistently to every matching file, handling the repetitive work of multi-file edits.

**/debug** provides systematic debugging. Describe the symptom and it follows a structured approach: reproduce the issue, isolate the cause, propose a fix, verify the fix, and check for regressions.

**/loop** enables iterative refinement. It makes a change, evaluates the result against criteria you define, and repeats until the output meets your standards. Useful for polish passes on documentation, UI styling, or performance optimization.

**/claude-api** integrates with the Anthropic API. It helps you write code that calls Claude's API, handling authentication, message formatting, streaming responses, and error handling correctly.

## Restricting Skill Access

Permission rules can control which skills are allowed to run. The `Skill(name)` permission type lets you allow or deny specific skills.

<CodeBlock
  code={`{
  "permissions": {
    "deny": [
      "Skill(deploy)"
    ]
  }
}`}
  lang="json"
  title="settings.json"
  caption="Denying the deploy skill prevents both user slash command and Claude auto-invocation."
/>

Enterprise administrators can use managed settings to enforce skill restrictions across the organization. This ensures that only approved skills run in production-sensitive environments.

## Best Practices

- **Keep skills focused on one task.** A skill that does one thing well is more reusable and easier to debug than a skill that tries to handle multiple scenarios.

- **Write clear descriptions for discoverability.** The description field is what users see in the slash command menu and what Claude uses to decide when to auto-invoke. Make it specific and action-oriented.

- **Use allowed-tools to limit scope.** A code review skill should not need write access. A documentation generator should not need to run shell commands. Restricting tools prevents unintended side effects.

- **Test skills with simple inputs first.** Before deploying a complex skill to your team, test it with straightforward cases to verify the instructions produce the expected behavior.

- **Use context:fork for expensive operations.** Skills that read many files or perform lengthy analysis should run in subagents to preserve your main session's context window.

- **Prefer project skills for team sharing.** Store skills in `.claude/skills/` and commit them to Git. This ensures every team member has the same skill set and updates propagate through normal version control.

- **Use dynamic injection sparingly.** The `!`command`` syntax is powerful but runs shell commands on every invocation. Keep injected commands fast and side-effect free.
